### Project Background & Problem Statement
During the physical design phase of VLSI development, macro placement is an NP-hard optimization problem heavily bottlenecked by sequential CPU calculations. Suboptimal placement degrades a chip’s Power, Performance, and Area (PPA) metrics. 

While pure Machine Learning placement models generate coordinates rapidly, they fundamentally fail to enforce strict zero-overlap physical design rules, resulting in illegal, unmanufacturable layouts. **RTLign** addresses this gap by acting as a drop-in heterogeneous accelerator. We hijack the standard EDA flow, replacing the traditional software macro placer with a high-speed ML Predictor coupled with a deterministic, zero-overlap RTL Hardware Legalizer.

### Pipeline Architecture
Our custom hardware-software co-design point tool integrates directly into the open-source OpenROAD physical design flow.

1. **Automated Orchestration:** A Python script utilizing the `subprocess` module acts as the master controller, providing a single-click automated execution of the entire pipeline.
2. **The Extraction (OpenROAD):** OpenROAD synthesizes a netlist and halts, dumping the chip's physical state to a Design Exchange Format (`.def`) file. Macro orientations are strictly locked to default (N) to prevent routing failure.
3. **The ML Predictor (Python):** Python parses the `.def` file. A Supervised ML model predicts approximate, wirelength-optimized coordinates (with expected overlaps) and formats them into a `.hex` memory file.
4. **The RTL Legalizer (Verilog):** A custom Verilog Simulated Annealing engine reads the `.hex` file. Operating at a low initialization temperature with a massive overlap penalty, it uses parallel combinational logic to physically push overlapping blocks apart. Upon completion, a Verilog testbench automatically exports the legalized coordinates to a new output file via `$fdisplay` / `$writememh`.
5. **The Injection & Signoff (OpenROAD):** Python rewrites the final Verilog coordinates back into the `.def` file. OpenROAD reads the modified file, renders the legalized floorplan natively in the OpenROAD GUI, and resumes execution to run routing and STA signoff.

### Tools & Technologies
* **Languages:** Python 3.x, Verilog HDL, TCL.
* **Frameworks:** OpenROAD (Physical Design EDA Suite), Scikit-Learn / PyTorch (ML Predictor).
* **Platforms:** Icarus Verilog (`iverilog`) for RTL simulation, Git/GitHub for version control.

### Datasets
* **Dataset 1 (Baseline):** Open-Source RISC-V Cores (e.g., PicoRV32) from the OpenROAD Project / GitHub. These serve as the raw, baseline inputs for our physical design pipeline, providing real-world connectivity graphs and macro dimensions.
* **Dataset 2 (Harvested):** 10,000+ highly optimized `.def` layouts generated by automating OpenROAD placement on Dataset 1 under randomized constraints (varying aspect ratios, utilization targets, and blockages). This will extract 10,000+ highly optimized .def layouts to train our Supervised ML Predictor.

### Project Timeline

* **Week 1–2:** Build the TCL/Python scripting pipeline to automate OpenROAD. Generate the thousands of highly optimized `.def` datasets required for supervised learning.
* **Week 3–4:** Train the Python ML Predictor on the extracted datasets. Write the Python parser to convert `.def` files into `.hex` format for the hardware initialization.
* **Week 5–6:** Design and simulate the parallel Verilog RTL SA Legalizer. Implement testbench file I/O for coordinate extraction and lock down the internal bit-widths for calculations to prevent integer overflow.
* **Week 7–8:** Build the Python subprocess orchestration script. Verify the native OpenROAD GUI rendering of the hardware-legalized layouts and ensure STA tools can successfully parse the modified DEF structure.
* **Week 9+:** Execute final benchmarking. To avoid software simulation bias, calculate theoretical hardware latency. Compare this against the pure software baseline, run final OpenROAD routing, and prepare defense documentation.
